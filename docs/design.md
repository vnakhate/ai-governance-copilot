## Problem Statement

Enterprises are drowning in AI tool adoption they can't see or govern. The 2026
discourse is blunt about it: the average org sees ~223 shadow-AI incidents/month,
~665 distinct GenAI tools show up across a typical enterprise, only ~40% bought
official subscriptions, only ~37% have any AI governance policy at all, and
shadow-AI breaches average $4.63M.

When a department wants to onboard a new AI tool — vendor-bought (e.g. an AI
notetaker) or internally built — there's no fast, trustworthy answer to "can we
use this, for this kind of data, in our jurisdiction, and what controls do we
need?" The decision either stalls in a slow governance-team intake process, or
the employee shrugs and uses the tool anyway. That gap is exactly where shadow
AI leaks.

This project builds a **governance copilot** on the Glean platform: an enterprise
assistant employees ask in plain language, which answers from live company policy,
produces a structured **risk verdict + required controls**, and captures the
onboarding decision into **Vanta** as the compliance system of record.

## What Makes This Cool

The "whoa" is the moment the copilot stops behaving like a search box and starts
behaving like a **risk analyst**:

> Employee: "Can we use Acme AI Notetaker on customer calls?"
> Copilot: "HIGH risk. Acme routes call audio to US servers, has no SOC 2, and no
> DPA is on file — that's a GDPR gap for EU customer data. Required controls before
> use: signed DPA, EU data residency, retention < 30 days. Onboard into Vanta with
> this risk captured? [Yes]"
> → Glean custom action writes a Vanta AI-vendor record (risk=HIGH, controls attached)
> → "Logged in Vanta as vendor #..., flagged HIGH, owner notified."

Three things make it distinct from vanilla Glean Assistant with policy PDFs indexed:
1. **Structured judgment, not retrieval** — it reasons over data-type × tool ×
   jurisdiction → a verdict and the exact controls, not a quoted paragraph.
2. **It acts** — the verdict lands in Vanta, a real system of record, instead of
   evaporating after the chat.
3. **It lives at the decision moment** — inside the assistant employees already use,
   not in a back-office GRC portal they avoid.

## Constraints

- Hackathon / demo timeline. Reliability on stage matters more than completeness.
- Built on a **real Glean instance** with Agent Builder (authentic, but introduces
  a live-platform dependency — see P3 and The Assignment).
- Vanta capture should *look and feel* real; live API is a stretch goal, not a gate.
- Target output is a working demo + this design doc. No production hardening expected.

## Premises (all confirmed by the user)

1. **The demo's center of gravity is the risk verdict, not retrieval.** Indexing
   policy PDFs is table stakes; the structured verdict (score + required controls,
   reading like a risk analyst) is the differentiator.
2. **The verdict needs a structured tool/vendor knowledge source, not just policy
   docs.** Policy says "no PII in unapproved tools"; it does not say "Acme Notetaker
   has no SOC 2 and routes audio to US." The demo therefore needs a curated set of
   ~8–12 tool profiles (capabilities, data residency, certs, approval status). This
   is the main build dependency.
3. **Hands-on Glean Agent Builder access exists before demo day.** The entire
   "real Glean agent" path assumes the ability to index a corpus and define agent
   instructions + a custom action. This is the single biggest feasibility risk.
4. **Aim at the decision moment, not a GRC portal.** The market (Trustible, ModelOp,
   OneTrust) builds intake/approval workflows for governance teams. Shadow AI leaks
   because employees make per-use-case decisions with no fast answer. The edge:
   governance inside the assistant employees already trust.
5. **Scope is held tight: a flawless 2–3 turn conversation.** One high-risk scenario,
   one clean verdict, one Vanta capture. Generalization across the curated tool set
   is in; sprawling feature surface is out.

## Landscape (three-layer synthesis)

- **Layer 1 (tried and true):** Mature AI governance platforms — AI inventories,
  intake/approval workflows, attribute-based risk scoring, vendor-doc analysis.
  Aimed at governance teams.
- **Layer 2 (current discourse):** The Shadow AI crisis. Recurring conclusion:
  *"Policy without enforcement or a good approved alternative is just a liability
  disclaimer."* Employees bypass governance because it's slow, invisible, and the
  sanctioned tools lag.
- **Layer 3 (EUREKA):** Everyone builds governance as a back-office GRC workflow,
  but shadow AI leaks at the **employee's per-use-case decision moment**, not at
  procurement. A copilot that gives an instant risk verdict *inside the assistant
  employees already use* attacks the failure at its actual source. Glean is uniquely
  positioned to be that assistant.

## Approaches Considered

### Approach A: Real agent, mocked Vanta (minimal viable / demo-safe)
Real Glean agent generalizes over ~10 curated tool profiles + policy corpus, outputs
the verdict. Vanta "capture" is a custom action returning a Vanta-styled record, no
live API. Effort M, Risk Low. Pro: hero is 100% real, never fails live. Con: Vanta
write is cosmetic.

### Approach B: Full loop, live Vanta write (ideal architecture)
Same agent, custom action calls the Vanta API to create a real AI-vendor risk record.
End-to-end real. Effort L, Risk Med-High. Pro: strongest "it actually did it" proof,
best product trajectory. Con: needs Vanta API access + sandbox; live API is the
classic demo killer.

### Approach C: Thin agent, fat data (creative / lateral)
Bet that verdict quality = data quality. Keep the agent prompt simple; invest in a
rich structured tool-risk knowledge base (~12 tools: data flows, certs, jurisdictions,
control templates). That same schema IS the Vanta record payload — authored once.
Effort M, Risk Low. Pro: exceptional verdicts, one schema serves verdict + Vanta. Con:
the wow is invisible (it's the data), needs narrative to make judges feel it.

## Recommended Approach

**Blend: A-spine + C-data, B if time.**

- Build on a **real Glean agent** — the verdict (the hero) is fully real and never
  depends on an external API on stage.
- Bake in **C's data discipline**: author the ~12 tool-risk profiles once, as
  structured records. They power both the verdict reasoning and the Vanta payload.
- Implement the **Vanta capture as a custom action that returns a Vanta-styled
  record (mock)**. It's the cheapest thing to fake and the easiest to make live later.
- **Stretch (B):** if Glean→Vanta API access lands early and is stable, flip the
  custom action from mock to a live Vanta write.

Rationale mapped to the goal (hackathon demo): live demos fail on external
dependencies, so keep the hero real and the fragile piece mocked. Authentic where
judges look (the verdict), safe where demos break (the live integration).

**Reliability anchor:** the hero scenario's answer is pinned (curated so it's
bulletproof). Generalization across the rest of the tool set is best-effort. If a
judge prods an off-set tool, the agent falls back to "MEDIUM — needs review" rather
than improvising a wrong verdict.

## Data & Schemas (build inputs)

Author the data first; the prompt is thin (Approach C). The *same* tool-risk profile
drives both the verdict and the Vanta record.

### Tool-risk profile (authored per tool)
```json
{
  "id": "acme-notetaker",
  "name": "Acme AI Notetaker",
  "vendor": "Acme Inc.",
  "capabilities": ["records call audio", "transcribes", "summarizes"],
  "data_residency": "US",
  "certs": [],
  "dpa_on_file": false,
  "retention_days": null,
  "approval_status": "unreviewed",
  "data_types_touched": ["call_audio", "customer_pii"],
  "control_templates": [
    {"gap": "no_dpa",           "control": "Signed DPA required"},
    {"gap": "non_eu_residency", "control": "EU data residency"},
    {"gap": "open_retention",   "control": "Retention < 30 days"}
  ]
}
```
`approval_status` ∈ `{unreviewed, approved, rejected}` (the rubric's LOW row keys on
`approved`). Required controls are NOT free-form: each profile carries
`control_templates[]` keyed to its gaps, so the verdict assembles its controls
deterministically.

### Data classification (a DELIVERABLE, indexed into Glean — not an open question)
Four levels: Public / Internal / Confidential / Regulated-PII. Map the demo inputs
explicitly so the verdict can reference them: `call_audio` + `customer_pii` ⇒
**Regulated-PII**.
**Why the hero is Regulated-PII (defend this if a judge pushes):** recordings/transcripts
of customer calls contain customer voice plus likely PII (names, account numbers, health
or financial details spoken aloud). Under GDPR/CCPA that is regulated personal data, so a
notetaker ingesting it touches Regulated-PII. This justifies the hero's HIGH rating rather
than leaving "is a notetaker PII?" as an unstated assumption.

### Risk rubric (encoded in the endpoint's `assess()` function — see Eng Decision 1A)
Computed in deterministic code, not the agent prompt. Fixed in eng review to make LOW
reachable for compliant PII vendors and to define the no-cert case (was a gap a judge
would poke):
| Condition | Verdict |
|---|---|
| Regulated-PII AND (no DPA OR non-EU residency) | HIGH |
| Regulated-PII AND DPA AND EU residency AND **no** cert | MEDIUM |
| Regulated-PII AND DPA AND EU residency AND cert AND approval_status = approved | LOW (approved) |
| Confidential/Internal AND approval_status = approved | LOW |
| Unknown tool / anything else / uncertain | MEDIUM — "needs review" |

### Verdict output template (the agent emits this shape every time)
```
Risk: <HIGH | MEDIUM | LOW>
Why: <data class> + <tool gap(s)> + <jurisdiction>
Required controls: <bulleted list pulled from control_templates>
Action: Onboard <tool> into Vanta with this risk captured? [Yes / No]
```

### Vanta record (the mock payload the custom action returns)
```json
{
  "record_id": "VND-2026-0142",
  "object": "ai_vendor_risk",
  "tool": "Acme AI Notetaker",
  "risk_level": "HIGH",
  "risk_justification": "Regulated-PII; no DPA; US residency (GDPR gap)",
  "required_controls": ["Signed DPA", "EU data residency", "Retention < 30 days"],
  "owner": "governance@company.com",
  "status": "flagged",
  "captured_at": "<ISO timestamp>"
}
```
**Profile → record mapping:** `name → tool`, verdict → `risk_level`,
`control_templates[].control → required_controls`. Record-only fields (`record_id`,
`owner`, `status`, `captured_at`, `risk_justification`) are generated at capture
time. Approach B maps these onto Vanta's existing vendor-risk / TPRM object.

## Engineering Decisions (locked in /plan-eng-review 2026-05-25 — supersede earlier approach notes)

Eng review reshaped the verdict architecture after grounding on Glean's own docs
(Glean does not host logic; a custom action requires a hosted HTTP endpoint; action
return values feed the agent's response).

- **1A — Verdict computed in the endpoint, not the prompt.** A pure
  `assess(profile, data_type, jurisdiction)` function encodes the rubric in testable
  code. Removes LLM-inconsistency risk under live judge prodding. Costs no extra infra
  because the endpoint must exist anyway for the Vanta write.
- **2A — Profiles in code, policy indexed.** Tool profiles live in the endpoint (drive
  the deterministic verdict). The AUP + data-classification docs are indexed into Glean
  so the agent retrieves and **cites real policy** — this is the honest answer to "what
  is Glean doing here?"
- **4A — Two execution actions.** `/assess` (read-only, no side effects, callable as soon
  as inputs are known) and `/capture` (writes the Vanta record, fires ONLY after a user
  "Yes"). Only `/capture` is turn-gated.
- **5A — Single shared `assess()`.** `/capture` re-runs the same function so the captured
  record is provably identical to the verdict the user approved. (Kept over the outside
  voice's objection — near-zero cost, future-proofs the live-Vanta path.)
- **3B — Hosting via ngrok→localhost** (user's call, for fast iteration). Mitigations:
  reserved ngrok domain (stable URL), a recorded backup clip, demo off a phone hotspot.

```
ENDPOINT CONTRACT
  POST /assess   { tool_id, data_type, jurisdiction } -> { risk_level, why, controls[] }   (read-only)
  POST /capture  { tool_id, data_type, jurisdiction } -> re-run assess() -> Vanta record   (gated write)
        shared:  assess(profile, data_type, jurisdiction)  <- single source of truth

AGENT FLOW (Glean runtime)
  user question --> [agent extracts tool/data/jurisdiction] --> POST /assess
       --> agent presents verdict + cites INDEXED policy --> "Onboard into Vanta? [Yes/No]"
       --(Yes)--> POST /capture --> "Logged in Vanta, flagged <risk>"
       --(No)---> stop, no write
```

### Failure modes (eng review)
| Failure | Tested? | Error handling? | User sees? | Verdict |
|---|---|---|---|---|
| ngrok tunnel drops mid-demo | manual | needs graceful msg | should be clear, not a hang | **critical gap** → recorded clip is plan A |
| Retrieval cites wrong/vague clause | smoke step 2 + eval | n/a | wrong-looking citation | **critical gap** → pre-test retrieval |
| Agent calls /capture before "Yes" | eval + manual | n/a | unwanted Vanta record | covered by 4A + eval |
| Double "Yes" / rapid resubmit | manual | /capture idempotent | duplicate record | covered if idempotent |
| Unknown/off-set tool | unit (assess default) | returns MEDIUM-needs-review | safe fallback | covered |
| Malformed profile at load | unit | fail fast at boot | n/a (pre-demo) | covered |

Two critical gaps (ngrok drop, retrieval quality) are the live-demo risks — both are
addressed by the reordered smoke test + recorded backup, not by code.

## Open Questions

- **Glean Agent Builder reality:** Can custom actions in your instance call an
  external endpoint, and what's the exact contract (request/response shape, auth)?
  This determines how the mock and the eventual live Vanta write are wired.
- **Vanta data model:** Does Vanta have a clean object for "AI vendor / AI tool risk,"
  or do we map onto the existing vendor-risk / TPRM object? (Affects B and the mock
  schema.)
- **Tool-risk data provenance:** The profiles are hand-authored for the demo. Are any
  real (your actual vendor list) or all synthetic? Synthetic is fine for a hackathon
  but say so if a judge asks.

## Success Criteria

- A **real Glean agent** answers the hero scenario end-to-end in under ~60 seconds,
  live, without an external-API failure.
- The verdict is **structured**: explicit risk level + named required controls +
  the specific reason (data type, jurisdiction, missing cert), not a generic policy
  quote.
- The copilot **generalizes across the tool set that exists on demo day**: a judge
  can name any tool in the curated set and get a coherent verdict. Demo-day minimum
  is 3 profiles (hero + 1 backup + 1 approved-LOW); ~12 is the stretch. Off-set tools
  return "MEDIUM — needs review," never a confidently-wrong verdict.
- The **Vanta capture** produces a believable AI-vendor risk record (mock for the
  demo; live Vanta write is a post-deadline stretch) with risk level + controls
  attached, shown on screen.
- The demo **tells the decision-moment story** clearly enough that an enterprise
  person in the room nods.

## Distribution Plan

For the hackathon, "distribution" is the demo itself: the agent published in your
Glean instance's Assistant/agent directory so it's invokable live. No app-store or
package distribution needed at this stage. If this graduates past the demo, the real
distribution channel is the Glean agent directory inside the customer's tenant, plus
a Glean→Vanta connector (the custom action hardened into a real integration).

## Next Steps (concrete build order)

1. **De-risk the platform first.** Run the smoke test in *The Assignment* below
   before anything else. Nothing in steps 2–7 matters if this fails.
2. **Author the data, not the prompt.** Write the 3 demo-minimum tool-risk profiles
   using the schema in *Data & Schemas* — one clearly HIGH (Acme Notetaker), one
   approved-LOW, one MEDIUM — each with `control_templates[]`. Expand toward ~12 as
   stretch.
3. **Write the policy corpus + classification.** A short AI Acceptable Use Policy +
   the 4-level data-classification doc (see *Data & Schemas*). Index both into Glean.
4. **Build the endpoint: `assess()` + `/assess` + `/capture`** (Eng Decisions 1A/4A/5A).
   The pure `assess()` function encodes the rubric in code; `/assess` returns the verdict
   read-only; `/capture` re-runs `assess()` and writes the Vanta record. Unit-test every
   rubric row here — this is what makes the demo bulletproof.
5. **Configure the agent + index policy** (2A). Agent extracts (tool, data_type,
   jurisdiction), calls `/assess`, presents the verdict citing the *indexed* policy, and
   on "Yes" calls `/capture`. Author the ~10-15 case eval (extraction, faithful
   presentation, turn-gating). For the demo, `/capture` returns a Vanta-styled record (mock).
6. **Rehearse the hero scenario; add one backup scenario.** Lock the exact 3-turn
   flow. Have a second tool ready in case a judge asks "what about X?"
7. **Stretch:** flip the Vanta action to a live API write if access + time allow.

## The Assignment

**Run a Glean Agent Builder smoke test before you build anything else.** The eng-review
outside voice reordered this: test the uncontrolled black boxes FIRST, turn-gating last,
because turn-gating is cheap config and the real unknowns are upstream of it. In order:

1. **Can Glean call your ngrok custom action AT ALL?** Stand up a trivial `/assess`
   behind ngrok, register the OpenAPI spec, confirm the agent invokes it and gets the
   return value into its answer. (Glean may enforce HTTPS/auth/spec validation and may
   cache the action manifest — find that out now.)
2. **Does retrieval cite cleanly?** Index one policy doc, ask a known question, confirm
   the agent retrieves and cites the *correct* clause — not a vaguely-related paragraph.
   This is the hero's real risk: the "grounded in real policy" wow lives or dies here.
3. **Does the structured verdict render legibly** in the Glean chat UI?
4. **Turn-gating:** confirm `/capture` (the write) fires ONLY after an explicit "Yes".

If steps 1-2 work, the real-Glean path is green. If retrieval is junk or Glean can't reach
the tunnel, you learn it in under an hour instead of on stage — and can fall back to the
recorded clip (the true plan A for the live network hop) or a standalone Glean-styled app.
