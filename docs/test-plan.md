Architecture locked this review: verdict computed in endpoint (1A); profiles in code +
policy indexed in Glean (2A); two actions /assess (read-only) + /capture (gated write) (4A);
single shared assess() function used by both (5A); endpoint hosted via ngrok→localhost (3B).

## Affected Pages/Routes (the hosted endpoint + the Glean agent)
- `POST /assess` — read-only. Input (tool_id, data_type, jurisdiction) → verdict (risk_level,
  why, required_controls). Must have ZERO side effects. Test it can be called repeatedly safely.
- `POST /capture` — gated write. Re-runs shared assess(), maps to Vanta record, writes
  (mock or live). Must fire ONLY after a user "Yes". Test the No path and double-Yes path.
- Glean agent (hosted) — extracts inputs, calls /assess, presents + cites indexed policy,
  on "Yes" calls /capture. Driven manually + via eval (Glean runtime is hard to automate in CI).

## Key Interactions to Verify
- Hero flow: ask high-risk tool → HIGH verdict + controls → "Onboard? Yes" → Vanta record created.
- Confirmation gating (★CRITICAL): user says "No" → /capture is NOT called, no Vanta record.
- Off-set tool: judge names a tool not in the profile set → agent returns "MEDIUM — needs
  review", never a confidently-wrong verdict.
- Approved-LOW tool: returns LOW; verify no false HIGH.
- Citation: the agent's "why" cites the indexed AI Acceptable Use Policy (proves Glean retrieval).

## Edge Cases
- Double "Yes" / rapid resubmit → /capture is idempotent (no duplicate Vanta record). Idempotency
  = running it twice produces the same single record, not two.
- Endpoint unreachable (ngrok tunnel drops, the 3B risk) → agent shows a graceful message,
  not a stack trace or silent hang.
- Ambiguous input ("can I use an AI tool?") → agent asks which tool instead of guessing.
- Malformed tool profile at load → fail fast at startup, not mid-demo.

## Critical Paths (must work on stage)
1. Hero: high-risk question → deterministic HIGH verdict → confirmed capture → Vanta record shown.
2. The "No" path: assessment without capture (proves the write is truly gated).
3. Off-set tool graceful fallback (the judge-prodding scenario).

## Unit tests (Vitest/Jest — endpoint logic, deterministic, cheap)
- assess(): HIGH / MEDIUM / LOW / DEFAULT-needs-review — one test per rubric row.
- assess(): control assembly from control_templates[] (controls match the gaps).
- loadProfiles(): malformed profile rejected at load.
- profile→Vanta mapping: every field maps correctly (name→tool, control_templates→required_controls).
- /assess integration: returns verdict, asserts no write occurred.
- /capture integration: writes via shared assess(); record == verdict shown; Vanta-unreachable → graceful error.

## Eval suite (agent behavior — scope 6A, ~10-15 cases)
- Extraction: varied phrasings → correct (tool, data_type, jurisdiction).
- Faithful presentation: agent relays endpoint verdict verbatim; no invented risk levels or controls.
- Turn-gating: agent never calls /capture before an explicit "Yes".

## Pre-demo smoke test (from The Assignment, do FIRST)
Validate turn-gated action invocation end-to-end: index one policy doc, stand up /assess + /capture
behind ngrok, confirm the agent calls /assess on the question and /capture ONLY after "Yes".
This is the single riskiest unknown — prove it before building the rest.
