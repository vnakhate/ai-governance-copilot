Intake workflow implemented · POST /request

Request a new AI tool

How an employee asks the Glean agent to onboard a tool that isn't in the registry yet. Turns the "needs review" dead-end into a tracked governance request.

01 Entry points

Two ways in

reactive

From a dead-end verdict

An /assess returns needs_review for an unknown tool, and the user says "request it" / "get it reviewed."

proactive

Opening with intent

"I want to use [tool]" / "can we get [tool] approved for my team?" — the agent recognizes onboarding intent up front.

02 Conversation

What the agent does

The agent collects only the fields it's missing, runs a best-effort provisional read, confirms, then writes a pending request. It captures and routes — it never adjudicates.

  1. Recognize the "request new tool" intent.
  2. Collect the intake fields (only what's missing):tool + vendor (or internal + team) · vendor URL · intended use · data types · department + requester · jurisdiction
  3. Provisionally assess on what's known, and list the unknowns a reviewer must resolve (open_questions).
  4. Confirm the summary: "Submit this request? [Yes / Edit]".
  5. On Yes → POST /request: write a PENDING record + audit('requested') + (optional) a governance ticket.
  6. Return the tracking ID: "Logged as VND-2026-0148 (PENDING). Routed to governance@. You'll be notified."
03 Lifecycle

State machine — the agent owns only requested

After submission, the governance team drives every transition in Vanta. The agent's job ends at capture + routing.

05 Plugging in

One new action, riding the existing engine

Decision: a dedicated requestNewTool action → POST /request. /assess = known tools, /capture = write a verdict, /request = new/unknown → pending. It reuses assess() for the provisional read but writes a request object, not a risk record. (Overloading /capture with a pending flag was rejected — it blurs "decision" vs "open request" and muddies the audit trail.)
