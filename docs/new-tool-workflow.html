The new-tool request workflow

When a tool isn't in the registry, the agent doesn't dead-end. It collects an intake, submits a pending request, and tells the requester exactly what the reviewer still needs.

01 The flow

Ask → intake → pending request

The agent's Step 5. It runs a provisional assessment, writes an ai_vendor_request (status pending_review), logs a requested audit event, and hands the requester the open questions. Governance adjudicates later, in Vanta.

02 The idea

The loop closes

The same tiered rubric that grades known tools now drives the intake: the attributes a tool is missing for its risk tier become the exact questions the request asks. Nothing is hand-maintained twice.

rubric output — missing_attributes
missingAttributes(profile, tier)
  • dpa_on_file
  • certs
  • bias_assessment
  • autonomy_level
  • … (tier-3 set)
→
request output — open_questions
openQuestions(missing)
  • Is a signed DPA on file?
  • SOC 2 / ISO 27001 / ISO 42001?
  • Has bias been assessed?
  • Suggests only, or takes actions?
  • …
One source of truth. The tier decides which attributes are required; whatever the requester hasn't answered is exactly what review must resolve. Higher risk → more questions, automatically.
