Architecture & design

AI Governance Copilot

A deterministic risk-verdict engine with a Glean chat client. An employee asks; the engine answers from policy + tool data; the decision is captured in Vanta and logged to an audit trail.

core principle → the verdict lives in a hosted endpoint, not the prompt. chat is client #1; a headless caller is client #2. one engine, many surfaces.
01 System architecture

Components and data flow

Two clients call one engine. The engine computes the verdict in code (deterministic), reads tool profiles, writes the decision to Vanta and to an append-only audit log. Glean's enterprise retrieval grounds the agent's policy citation — the one thing only Glean does.

02 Components

What each piece does

endpoint/index.mjs 8 tests

Zero-dependency Node server. The whole engine: rubric, routes, audit, Vanta mapping.

  • assess() — verdict from profile × data × jurisdiction
  • /assess — read-only verdict · /capture — gated write
  • recordAudit() / readAudit() — append-only trail · /audit
  • toVantaRecord() — profile+verdict → Vanta shape

profiles.json

Hand-authored tool risk data — capabilities, residency, certs, DPA, retention, approval, control templates.

  • 3 demo tools (HIGH / MEDIUM / approved-LOW)

policy/*.md

Indexed into Glean for grounded citation. AUP §3.2 = the regulated-PII clause the verdict cites.

  • aup.md · data-classification.md

agent/instructions.md

The Glean Agent Builder prompt: extract inputs → call /assess → render verdict + cite → gated /capture.

  • eval-cases.md — 16 behavior cases

scripts/demo-headless.mjs

Calls the same engine headlessly — proves client #2 gets an identical verdict + Vanta record.

endpoint/glean-action.yaml

OpenAPI spec to register /assess + /capture as Glean custom actions.

03 The verdict engine

Deterministic rubric — assess()

Computed in code, identical every run, unit-tested per row. The data class comes from the data type the employee wants to use; the gaps come from the tool profile; the jurisdiction decides residency.

04 Request flow

Ask → verdict → gated capture

The signature flow: /assess is read-only and always safe; /capture writes to Vanta and fires only after an explicit "Yes". Both call the same assess() (so the record matches the verdict) and both append to the audit trail.

sequenceDiagram
  actor E as Employee
  participant G as Glean agent
  participant EP as Endpoint
  participant V as Vanta
  E->>G: Can we use Acme on customer calls?
  G->>EP: POST /assess {tool, data, jurisdiction}
  Note over EP: assess() + audit(proceeded null)
  EP-->>G: HIGH + required controls
  G-->>E: verdict card + cite AUP 3.2 + [Yes/No]
  E->>G: Yes
  G->>EP: POST /capture
  Note over EP: assess() again (5A) + audit(proceeded true)
  EP->>V: create AI-vendor record
  EP-->>G: record_id, flagged
  G-->>E: Logged in Vanta, owner notified
If the employee says No, /capture is never called — the audit row shows proceeded = null
